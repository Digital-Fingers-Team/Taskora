# Taskora — خطة البناء بالمراحل

> نظام تشغيل للأعمال المتكررة: AI بينفّذ، خبير بشري بيراجع، وكل تنفيذ بيغذّي المنصة.

**القاعدة الحاكمة:** كل مرحلة تطلّع حاجة شغّالة (بتدفع أو بتثبت حاجة)، وتبني على اللي قبلها من غير ما ترجع تهدّ شغل. في الآخر تبقى لمّيت كل حاجة اتفقنا عليها — بالتدريج مش مرة واحدة.

**فلسفة الترتيب:**
- المراحل **0–3** → تثبت إن الفكرة بتدفع قبل ما تبني تعقيد.
- المراحل **4–6** → تبني الـ Moat (السمعة + تعلّم الكروت).
- المراحل **7–9** → تجهّزك لـ B2B والسوق.
- المرحلة **10** → الرؤية الكاملة، آخر حاجة لأنها الأغلى والأعقد.

## المرحلة 0 — الأساس (Foundation)

### ليه دلوقتي؟
ده اللي لو عملته غلط هتدفع تمنه في كل مرحلة بعده. الأساس مش feature بيتشاف، لكنه اللي بيخلّي كل feature بعده سهل أو جحيم. أهم نقطة هنا: تصمّم وأنت فاهم إن المنتج النهائي ضخم، حتى لو اللي بتبنيه دلوقتي بسيط.

### بتبني إيه بالظبط؟
- هيكل الـ Monorepo والخدمات.
- نظام الدخول والأدوار والمنظمات.
- نموذج الداتا الأساسي (مصمّم وهو فاهم إن الكارت = Blueprint).
- الأساس التقني اللي كل حاجة هتقعد عليه.

### المكونات التقنية
- **Monorepo:** Turborepo بـ 3 packages على الأقل — `web` (Next.js 14)، `api` (NestJS)، `shared` (types مشتركة + Zod schemas).
- **النواة:** NestJS + PostgreSQL + Prisma + Redis.
- **Auth:** من أول يوم بمفهوم **Organizations / Members / Roles** — مش "Company" لوحدها. ده الفرق اللي عدّاه الوصف الأول وصح إن GPT صحّحه.
  - `User` ممكن يكون عضو في أكتر من Organization.
  - كل عضوية ليها Role (Owner / Admin / Member / Operator).
  - Permissions مبنية على الـ Role من البداية حتى لو بسيطة.
- **i18n:** عربي/إنجليزي من الأول (next-intl). متأجّلهاش لأن إضافتها بعدين بتلمس كل component.

### تخلص بإيه (Definition of Done)
- تقدر تسجّل، تعمل Organization، تضيف أعضاء بأدوار، وكل واحد يشوف اللي مسموح له بس.
- مفيش كروت ولا مهام لسه — ده مظبوط، إحنا بنبني الأرض.

### فخاخ تتجنبها
- متبدأش بـ "Company" واحدة وتقول هكبّرها بعدين — إعادة هيكلة الـ multi-tenancy من أصعب الـ migrations.
- متبالغش في الـ permissions دلوقتي. RBAC بسيط بـ enum كفاية. الـ granular permissions تيجي في مرحلة 7.

## المرحلة 1 — الكارت كـ Blueprint (القلب)

### ليه دلوقتي؟
ده **المنتج الحقيقي**. كل اللي بعده وسيلة لتشغيله وتطويره وبيعه. لازم تثبّت مفهوم الكارت صح من البداية، لأن لو بنيته كـ "خطوات نصّية" هتعيد كتابته كله في مرحلة 6. اعتبره من اليوم الأول **Runtime Object** قابل للتطوّر — حتى لو لسه مش بتشغّله.

### بتبني إيه بالظبط؟
- تعريف الـ `Task Blueprint` بمكوناته الكاملة كـ schema.
- CRUD كامل للكروت (إنشاء يدوي، لسه من غير AI).
- صفحة الكارت اللي بتعرض كل ده.

### المكونات التقنية
بنية الـ Blueprint (خزّنها كـ JSONB مرن في حقول منفصلة، مش نص واحد):
- **Metadata:** عنوان، عمودي (vertical)، وصف، سبب التنفيذ، مستوى صعوبة، وقت متوقع، مهارات مطلوبة.
- **Inputs Schema:** تعريف المدخلات المطلوبة (نوع كل مدخل، إجباري/اختياري) — استخدم JSON Schema أو Zod.
- **Steps / Workflow:** دلوقتي مصفوفة خطوات منظّمة (كل خطوة: عنوان، وصف، أداة، مخرج متوقع). **صمّمها بحيث تستوعب بعدين** عُقد شرطية وحلقات — يعني array of typed nodes، مش array of strings.
- **Tools:** قايمة الأدوات المستخدمة (مجرد أسماء/مراجع دلوقتي، الـ Connectors الحقيقية في مرحلة 8).
- **Expected Output:** شكل المخرج النهائي.
- **Common Mistakes:** الأخطاء الشائعة.
- **AI Instructions** و **Human Instructions** منفصلين — ده مهم، لأن اللي بتقوله للـ AI غير اللي بتقوله للإنسان.

### تخلص بإيه
- تقدر تعمل كارت كامل بإيدك من الواجهة، تحفظه، وتفتح صفحته وتشوف كل مكوناته معروضة بشكل محترم.
- ده أصلك. لو وقفت هنا، يبقى عندك "محرّر SOP ذكي" — حاجة ليها قيمة لوحدها.

### فخاخ تتجنبها
- **الفخ الأكبر:** تبني الكارت كـ Workflow Engine حقيقي زي n8n/Temporal دلوقتي. ده شركة لوحدها وأخدت سنين تتبني. أنت محتاج JSON schema مرن بس، مش محرّك تنفيذ. الرؤية صح، التوقيت غلط.
- متعملش الـ steps حقل نصّي واحد. خليها structured من الأول عشان مرحلة 6 (التطوّر) تبقى ممكنة.

## المرحلة 2 — دورة المهمة (Task Lifecycle)

### ليه دلوقتي؟
دي الحلقة اللي بتثبت إن **الفكرة بتدفع**. لحد دلوقتي عندك كروت بس مفيش حركة. هنا الشركة بتطلب، المنفّذ بينفّذ، والشركة بتراجع. لو الحلقة دي اشتغلت يدوي، يبقى عندك خدمة حقيقية حتى من غير AI.

### بتبني إيه بالظبط؟
- إنشاء `Task` من كارت موجود.
- دورة حالات المهمة.
- مساحة عمل المنفّذ.
- صفحة مراجعة الشركة.
- **سجل التنفيذ** — ده أهم حاجة هنا للمستقبل.

### المكونات التقنية
- **Task model:** `card_id`, `organization_id`, `operator_id`, `status`, `inputs`, `output`.
- **State machine للحالات:** `Draft → Assigned → In Progress → In Review → Completed` (و `Rejected / Revision Requested`). استخدم state machine حقيقي مش string عشوائي، عشان متسمحش بانتقالات غلط.
- **Operator Workspace:** الكارت على جنب (read-only reference)، مساحة التنفيذ على جنب، زرار رفع النتيجة.
- **Review page:** الشركة تشوف المخرج، تقبل / تطلب تعديل / تقيّم.
- **Execution Log:** أهم جدول للمستقبل. لكل تنفيذ سجّل: الوقت المستغرق، فين المنفّذ اتلخبط، أي ملاحظات. ده الوقود اللي هيغذّي تعلّم الكروت في مرحلة 6 — لو مسجّلتوش من دلوقتي هتفقد البيانات للأبد.

### تخلص بإيه
- شركة تطلب مهمة → منفّذ ينفّذها → شركة تراجع وتقيّم. الحلقة كاملة، يدوي بالكامل.
- عندك أول execution logs حقيقية بتتراكم.

### فخاخ تتجنبها
- متأجّلش الـ Execution Log لمرحلة 6. لازم يبدأ يجمع داتا من أول تنفيذ.
- خلّي الحالات state machine صريحة، مش if-else على string.

## المرحلة 3 — الفلوس (Payments & Billing)

### ليه دلوقتي؟
دلوقتي بالظبط — بعد ما الحلقة بتشتغل يدوي. الفلوس بتحوّل "demo بيشتغل" لـ "بزنس بيتختبر بفلوس حقيقية". قبل كده مش محتاجها، بعد كده متأخر.

### بتبني إيه بالظبط؟
- توزيع الفلوس على كل مهمة.
- اشتراكات الشركات.
- لوحة الفوترة.

### المكونات التقنية
- **Transactions model:** `task_id`, `amount`, `platform_fee`, `operator_payout`. (مثال: الشركة 100$، المنصة 20$، المنفّذ 80$).
- **بوابة دفع:** Stripe أو ما يناسب منطقتك. ابدأ بـ payments بسيطة قبل الـ payouts التلقائية.
- **Subscription tiers** للشركات (Free / Pro / Business) — حتى لو الفروق بسيطة دلوقتي.
- **لوحة فوترة:** الشركة تشوف صرفها، المنفّذ يشوف أرباحه.

### تخلص بإيه
- فلوس حقيقية بتدخل وتتوزّع صح.
- هنا تقدر تروح لـ 5 شركات وتختبر إنهم بيدفعوا فعلاً — أهم validation في المشروع كله.

### فخاخ تتجنبها
- متعقّدش الـ payouts التلقائية للمنفّذين في أول نسخة. ممكن تكون يدوية في البداية لـ 5–10 منفّذين.
- خد بالك من ضرائب/تحويلات عبر الدول بدري لو منفّذينك عالميين — بس متبنيش حل كامل دلوقتي، بس اعرف إنه جاي.

## المرحلة 4 — السمعة والـ Matching

### ليه دلوقتي؟
بعد ما يبقى عندك **تنفيذات حقيقية** تتبنى عليها السمعة. السمعة بدون بيانات تنفيذ حقيقية = أرقام مزيّفة. دلوقتي عندك logs كفاية تبني عليها profile حقيقي وmatching ذكي.

### بتبني إيه بالظبط؟
- نظام سمعة مبني على السجل الفعلي.
- ملف مهني للمنفّذ.
- مطابقة دلالية بين المهام والمنفّذين.

### المكونات التقنية
- **`reputation_score`:** محسوب من بيانات حقيقية — نسبة النجاح، سرعة التنفيذ، تقييمات الشركات، عدد مرات تنفيذ نفس الكارت.
- **Operator Profile:** التخصصات (مستنتجة من نوع الكروت اللي نفّذها)، نسبة النجاح، السرعة، السجل.
- **pgvector للـ Matching الدلالي:** embeddings للكروت + للمنفّذين (مهاراتهم وسجلهم)، والنظام يقترح الكارت للمنفّذ الأنسب. ده اللي بيحوّل "بحث عن شغل" لـ "النظام بيجيبلك الشغل المناسب".

### تخلص بإيه
- المنفّذ الجديد اللي نفّذ 50 كارت بحث بنجاح بيظهر كمتخصص في البحث — تلقائيًا.
- النظام بيقترح كل مهمة للشخص الصح.

### فخاخ تتجنبها
- متبنيش embeddings معقّدة من غير داتا. لو لسه التنفيذات قليلة، ابدأ بـ matching بسيط بالـ tags وكبّره للـ vector بعدين.
- خلّي السمعة شفافة — المنفّذ يفهم ليه نقطته كده.

## المرحلة 5 — طبقة الـ AI (نسخة واقعية)

### ليه دلوقتي؟
أخّرناها لحد هنا عن قصد. لأن الـ AI من غير حلقة شغّالة وداتا = تعقيد بدري على فاضي. دلوقتي عندك كروت ناضجة، تنفيذات، ومراجعة — الـ AI يقدر يضيف قيمة فعلية. **القاعدة: agent واحد كويس + مراجعة بشرية. مش أربع وكلاء.**

### بتبني إيه بالظبط؟
- AI بيحوّل كلام الشركة → كارت.
- AI بينفّذ الجزء الميكانيكي، الإنسان يراجع.
- شات مساعد أثناء التنفيذ.

### المكونات التقنية
- **Card Generation:** الشركة تكتب طلبها بالكلام العادي، الـ AI يحوّله لـ Blueprint مبدئي (أو يربطه بكارت موجود).
- **Execution Assist:** الـ AI ينفّذ الجزء القابل للأتمتة (بحث، تجميع، صياغة)، والمنفّذ يراجع ويصحّح ويتدخّل في الحكم.
- **In-task Chat:** المنفّذ يسأل الـ AI ويطلب تحسينات أثناء الشغل.
- **Background jobs:** BullMQ + Redis لتشغيل مهام الـ AI في الخلفية (مش blocking للـ request).
- **AI:** Claude API كطبقة خدمة منفصلة في الـ NestJS (service مستقلة، مش متناثرة في الكود).

### تخلص بإيه
- AI فعلي بيشتغل في الحلقة، بس بانضباط — بيرفع الإنتاجية من غير ما يعقّد المعمارية.

### فخاخ تتجنبها
- **متبنيش الـ Agent Orchestrator بأربع وكلاء دلوقتي.** ده مكلّف، بطيء، ومعقّد. agent واحد كويس بيدّي 80% من القيمة بـ 20% من التعقيد. الترقية في مرحلة 10 لما الحجم يطلبها.
- خلّي الـ AI service معزولة عشان تقدر تغيّر/تكبّر من غير ما تلمس باقي النظام.

## المرحلة 6 — تطوّر الكروت (Learning + Versioning)

### ليه دلوقتي؟
**هنا الـ Moat بيشتغل فعليًا.** GPT كان محق 100%: القيمة مش في بيع الكارت، القيمة إن الكارت **بيتعلّم**. version 14 أحسن من version 1 بعشر مرات. دلوقتي عندك execution logs + AI، فتقدر تقفل الحلقة: تنفيذ → تحليل → تحسين → نسخة أذكى.

### بتبني إيه بالظبط؟
- نظام نسخ زي Git للكروت.
- تحليل سجل التنفيذ.
- تحديث الكروت تلقائيًا للأفضل.

### المكونات التقنية
- **Versioning زي Git:** `Draft / Published / Archived` + `Rollback` + `Diff` بين النسخ.
- **`card_versions` table:** كل نسخة بتخزّن السبب (ليه اتغيّرت) + إيه اللي اتغيّر.
- **Learning loop:** الـ AI بياخد الـ Execution Logs (فين الناس بتتلخبط، الأخطاء المتكررة، الوقت)، ويقترح تحسين على الكارت → نسخة جديدة بتترفع كـ Draft للمراجعة قبل ما تتنشر.
- **Metrics لكل نسخة:** نسبة نجاح كل version عشان تقارن وتعرف التحسين حقيقي ولا لأ.

### تخلص بإيه
- الكارت بيتعلّم بمرور الوقت. version 5 قابل للقياس إنه أحسن من version 1.
- ده الأصل اللي محدش يقدر ينسخه — لأنه مبني على بياناتك أنت.

### فخاخ تتجنبها
- متخليش الـ AI ينشر نسخة تلقائيًا من غير مراجعة. النسخة الجديدة تيجي Draft → بشري يوافق → Published.
- لازم الـ Diff يكون واضح، عشان تفهم الكارت اتغيّر إزاي وليه.

## المرحلة 7 — جاهزية B2B (Enterprise Readiness)

### ليه دلوقتي؟
دلوقتي عندك منتج بيشتغل وبيتعلّم. عشان الشركات الجادة **تثق وتدمج**، محتاجة الطبقة الـ enterprise. دي مش features للمستخدم، دي شروط دخول السوق الكبير.

### بتبني إيه بالظبط؟
- API عام للتكامل.
- Webhooks.
- سجلات تدقيق.
- نظام إشعارات.

### المكونات التقنية
- **Public API:** `POST /tasks`, `POST /cards`, `GET /executions`, `POST /review` — موثّق (OpenAPI/Swagger) عشان الشركات تدمجه في أنظمتها.
- **Webhooks:** `Task Completed` / `Task Failed` / `Review Requested` — عشان النظام يبلّغ أنظمة الشركة بالأحداث.
- **Audit Logs:** مين غيّر إيه، إمتى، القيمة القديمة والجديدة. أي منصة B2B جادة لازم يكون فيها ده.
- **Notifications / Events:** `Task Assigned`, `Review Needed`, `Payment Done`, `Card Updated`, `Execution Failed` — نظام أحداث داخلي بيغذّي الإشعارات والـ webhooks.

### تخلص بإيه
- شركة تقدر تدمج Taskora في نظامها برمجيًا، وتثق فيه لأن كل حاجة متتبّعة ومُبلّغ عنها.

### فخاخ تتجنبها
- متعملش API بدري قبل ما النموذج يستقر — هتكسر العملاء بتغييرات كل أسبوع. عشان كده هي في مرحلة 7 مش 1، عكس ما اقترح GPT. (الفكرة صح، التوقيت اللي قاله غلط.)
- نسّخ الـ API (v1) من البداية عشان تقدر تطوّر من غير ما تكسر.

## المرحلة 8 — الأدوات والمعرفة (Connectors + Knowledge)

### ليه دلوقتي؟
لحد دلوقتي الكارت بيقول "افتح Google Sheets". هنا بيتحوّل لـ "اتصل بـ Google Sheets، اقرأ صف، اكتب صف" — يعني بينفّذ في العالم الحقيقي فعليًا. ده بيرفع قيمة الكارت من تعليمات لتنفيذ.

### بتبني إيه بالظبط؟
- إدارة بيانات اعتماد آمنة.
- موصّلات أدوات حقيقية.
- قاعدة معرفة لكل كارت.

### المكونات التقنية
- **Secrets Manager:** Encrypted Credentials — الكروت هتحتاج مفاتيح Google / Slack / Notion / HubSpot / OpenAI... لازم تتخزّن مشفّرة بأمان (مش plain text أبدًا).
- **Tool Connectors:** بدل نص "افتح Sheets" → `Google Sheets Connector → Action → Read Row / Write Row`. كل connector بيعرّف actions منظّمة.
- **Knowledge Base لكل كارت:** ربط الكارت بـ Files / Links / PDFs / Videos / SOPs / Policies، والـ AI يعمل **RAG** عليها أثناء التنفيذ — عشان ينفّذ بفهم سياق الشركة مش بشكل عام.

### تخلص بإيه
- الكارت بيتكلم مع أدوات حقيقية وبيستند لمعرفة حقيقية خاصة بالشركة.

### فخاخ تتجنبها
- الأمان هنا مش رفاهية. تسريب credentials عميل = نهاية الثقة. شفّر، اعزل، sandbox.
- ابدأ بـ 2–3 connectors اللي عمودياتك محتاجاها فعلًا (غالبًا Sheets + Slack)، مش 10 مرة واحدة.

## المرحلة 9 — السوق والجودة (Marketplace + Quality Gates)

### ليه دلوقتي؟
دلوقتي بس — بعد ما يبقى عندك **كروت ناضجة** (اتعلّمت عبر إصدارات) تستاهل تتعرض وتتباع. لو عملت Marketplace بدري هتعرض كروت فجّة وتحرق السوق. الجودة قبل العرض.

### بتبني إيه بالظبط؟
- سوق الكروت الجاهزة.
- صلاحيات الكروت.
- بوابات اختبار قبل النشر.
- ضمان جودة المنفّذين.

### المكونات التقنية
- **Card Marketplace:** تصفّح حسب العمودي، معاينة كارت (الخطوات، متوسط الوقت، نسبة النجاح، عدد التنفيذات)، شراء/تفعيل.
- **Card Permissions:** `Private / Team / Organization / Marketplace / Premium` — مش كل كارت عام.
- **Card Testing:** `Run Simulation → Pass → Publish`. مفيش كارت ينزل السوق من غير ما يعدّي simulation.
- **Quality Gates للمنفّذين:** اختبارات قبول قبل الوصول لمهام معيّنة + نظام مستويات (levels). ده بيحلّ مشكلة "جودة التنفيذ" اللي هي أكبر خطر في الفكرة كلها.

### تخلص بإيه
- سوق كروت ناضجة بجودة مضمونة، ومنفّذين متفاوتين بمستويات واضحة.

### فخاخ تتجنبها
- متفتحش السوق للعامة قبل ما الجودة تبقى مضبوطة — أول انطباع سيء بيقتل marketplace.
- الـ Simulation مش شكليّة. لازم تختبر الكارت فعلًا قبل النشر.

## المرحلة 10 — تعدّد الوكلاء (Agent Orchestration)

### ليه دلوقتي؟
آخر حاجة عن قصد. دي الرؤية الكاملة اللي GPT وصفها — بس وصلتلها بالتدريج بعد ما كل خطوة قبلها كانت بتدفع لوحدها. ترقّي الـ AI من agent واحد لفريق وكلاء **بس لما الحجم والتعقيد يطلبوا ده فعلًا**، مش قبلها.

### بتبني إيه بالظبط؟
- ترقية طبقة الـ AI لأوركسترا وكلاء.
- محرّك تعلّم بيغذّي المنصة كلها.

### المكونات التقنية
- **Agent Orchestrator:** `Planner → Research → Execute → QA → Human → Delivery`. الـ AI نفسه بقى فريق متخصص مش موديل واحد.
- **Learning Engine مركزي:** كل تنفيذ في المنصة بيغذّي تحسين الكروت، السمعة، والـ matching — مش بس الكارت اللي اتنفّذ. ده اللي بيخلّي المنصة كلها بتتحسّن مع كل حركة.

### تخلص بإيه
- الرؤية الكاملة: Dashboard → Task/Card Engine → Version Engine → Execution Engine → Agent Orchestrator → Human Review → Learning Engine → Metrics.
- وصلتلها وأنت كل مرحلة في الطريق كانت قائمة بذاتها وبتدفع.

### فخاخ تتجنبها
- متستعجلش المرحلة دي. أكتر مرحلة ممكن تستنزف وقتك من غير عائد لو دخلتها بدري.
- كل agent إضافي = تكلفة + بطء + نقطة فشل. ضيف agent بس لما تثبت إن الحاجة ليه حقيقية بالبيانات.

## الستاك المقترح (متماشي مع خبرتك)

- **Frontend:** Next.js 14 (App Router) + Tailwind + i18n عربي/إنجليزي.
- **Backend:** NestJS مع طبقة خدمات معزولة للـ AI.
- **DB:** PostgreSQL + pgvector (للـ matching الدلالي).
- **Queue:** BullMQ + Redis (مهام الـ AI في الخلفية).
- **AI:** Claude API لطبقة إنشاء/مراجعة/تحديث الكروت.
- **Monorepo:** Turborepo (web + api + shared types).

> أغلب الستاك ده عملته قريب في PriceLens، فمش هتبدأ من الصفر.

## ملاحظة معمارية واحدة تحكم القرارات كلها

خلّي **محرّك الكروت (Card Engine)** خدمة مستقلة من أول يوم — هو الأصل الدائم بتاعك. السوق والمنصة والـ UI ممكن يتغيّروا، بس محرّك تخزين وتطوّر الكروت هو القيمة اللي بتتراكم ويصعب نسخها. متخليهوش متشابك مع كود الـ Marketplace أبدًا.
